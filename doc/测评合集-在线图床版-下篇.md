## 13. 阶段七：Windows 上位机与 Claude Code 状态联动

第 12 阶段完成后，开发板已经具备稳定的显示、动画、DHT11、LED 和串口协议能力。本阶段继续向“Claude 状态监视器”推进：在 PC 端实现一个 Windows 上位机，用于连接 Claude Code、串口和开发板 LCD，使 Claude 的工作状态、模型信息、上下文使用率、费用和权限审批状态能够显示到 GD32C231C-EVAL 上。

本阶段不是让开发板直接访问网络或调用 Claude API，而是采用更稳妥的三段式架构：

```text
Claude Code
  ↓ Hooks / statusLine
Windows 上位机
  ↓ USART0 115200 8N1
GD32C231C-EVAL LCD / LED / PA0 / PA4
```

这样设计的原因很直接：GD32C231C8 只有 64 KB Flash 和 12 KB SRAM，不适合直接承担 HTTPS、JSON API、证书、复杂 UI 以及模型状态解析。Claude Code 本身运行在 PC 上，上位机天然更适合处理 Hooks、statusLine、权限确认和串口转发，开发板只负责显示和交互。

![alt text](https://raw.githubusercontent.com/318973613/gd32c231c-claude-monitor/master/doc/image-26.png)

### 13.1 上位机工程结构

本阶段新增两个 PC 端工程：

```text
tools\claude_monitor_desktop
tools\claude_monitor_hook
```

其中：

- `ClaudeBoardMonitor.exe`：图形化上位机，负责串口连接、状态显示、审批按钮和本地 HTTP 服务
- `ClaudeBoardHook.exe`：供 Claude Code Hooks/statusLine 调用的控制台程序
- `ClaudeBoardMonitor-win-x64.zip`：最终可分发压缩包，其他 Windows x64 电脑解压后即可使用

发布目录：

```text
tools\claude_monitor_desktop\publish\win-x64
```

压缩包：

```text
tools\claude_monitor_desktop\ClaudeBoardMonitor-win-x64.zip
```

上位机采用自包含发布方式，最终用户不需要额外安装 .NET、Python 或 Node.js。图形界面为中文界面，面向实际演示和分享使用。

![alt text](https://raw.githubusercontent.com/318973613/gd32c231c-claude-monitor/master/doc/image-27.png)

### 13.2 Claude Code Hooks 与 statusLine

Claude Code 提供 Hooks 和 statusLine 能力。Hooks 用于捕获用户输入、工具调用、权限请求、停止事件等；statusLine 用于周期性输出当前模型、上下文使用率、费用等运行信息。

上位机点击“安装 / 更新 Hooks”后，会备份并修改：

```text
%USERPROFILE%\.claude\settings.json
```

写入内容包括：

- `UserPromptSubmit`
- `PreToolUse`
- `PermissionRequest`
- `PostToolUse`
- `PostToolUseFailure`
- `Notification`
- `Stop`
- `statusLine`

其中 `statusLine` 指向：

```text
"ClaudeBoardHook.exe" --statusline
```

`PreToolUse` 使用 `*` 匹配，这样普通工具调用也能更新开发板状态；真正需要审批的高风险工具仍由 HookClient 进行判断，例如：

```text
Bash
Write
Edit
MultiEdit
NotebookEdit
```

![alt text](https://raw.githubusercontent.com/318973613/gd32c231c-claude-monitor/master/doc/image-28.png)。

### 13.3 PC 与开发板串口协议

上位机和开发板之间仍使用 ASCII 行协议，便于串口助手调试，也便于排查问题。

PC 发给开发板：

```text
AI=IDLE
AI=WORKING
AI=TOOL:<tool>
AI=NATIVE_APPROVE:<tool>
AI=WAIT_APPROVE:<tool>
AI=APPROVED
AI=DENIED
AI=DONE
AI=ERROR:<reason>
TIME=HH:MM:SS
MODEL=<model>
CTX=<used_percent>,<remaining_percent>
RATE=<used_percent>,<remaining_percent>
COST=<usd_cents>
```

开发板发给 PC：

```text
BTN=APPROVE
BTN=DENY
```

其中 `CTX` 表示上下文使用率，`COST` 使用美分整数传输。例如 Claude Code 当前会话费用为 `$0.13`，上位机会下发：

```text
COST=13
```

开发板屏幕显示为：

```text
USD 000.13
```

这样避免在 Cortex-M23 端处理浮点数和复杂格式化。

### 13.4 开发板工作界面

第 08 工程在空闲和工作两种状态下采用不同布局。

空闲状态保留前一阶段较稳定的界面：时间、DHT11 温湿度、计数器和空闲小人动画。工作状态则隐藏温湿度，避免 240x320 小屏内容拥挤，改为显示 Claude 相关信息：

- 当前状态：`WORK`、`WAIT`、`TERM`、`ALLOW`、`DENY`、`ERROR`
- 当前工具：如 `BASH`
- 模型：如 `MIMO-V2.5-PRO`
- 上下文：如 `CTX 012%`
- 费用：如 `USD 000.13`

工作状态的屏幕信息来自上位机和 Claude Code statusLine。若某项字段暂时没有提供，则显示 `---`，不会伪造数据。

### 13.5 PA0 / PA4 权限审批

本阶段将开发板按键用于 Claude 权限审批：

| 按键 | 开发板引脚 | 功能 |
| --- | --- | --- |
| PA0 | Wakeup | 同意 |
| PA4 | Userkey | 拒绝 |

当 Claude Code 触发需要审批的工具调用时，如果上位机处于“外部审批模式”，Claude 会等待上位机返回决定。此时可以通过三种方式审批：

1. 按开发板 PA0 同意
2. 按开发板 PA4 拒绝
3. 在上位机 GUI 中点击“同意 / 拒绝”

为了提升按键体验，固件端将按键扫描周期调整为 `10 ms`，上位机端增加了 2.5 秒预按缓存。也就是说，如果用户稍早按下 PA0/PA4，而审批请求马上到达，上位机会缓存这次按键并自动应用到最近的审批请求上，避免必须连续按很多次。

当外部审批模式关闭时，Claude 会使用原生终端确认。此时上位机和开发板会提示进入终端确认状态，开发板显示 `TERM 123`，表示需要回到 Claude Code 终端中按 `1/2/3`，PA0/PA4 不会控制这一次原生提示。

### 13.6 LED 状态设计

四个板载 LED 不再只是简单流水灯，而是作为辅助状态提示：

| 状态 | LED 效果 |
| --- | --- |
| 空闲 | 慢心跳 |
| 工作中 / 工具调用 | 流水灯 |
| Claude 原生终端确认 | 对角交替闪烁 |
| 等待外部审批 | 左右交替闪烁 |
| 已同意 | 全亮脉冲 |
| 已拒绝 | 单灯警示 |
| 错误 | 全灯闪烁 |

LCD 可以显示详细信息，LED 则适合从远处快速判断状态。例如看到左右交替闪烁，就知道当前不是普通运行，而是需要用户审批。

## 14. 本项目当前完整链路

到目前为止，项目已经从基础外设验证推进到完整的 Claude 状态显示链路：

```text
Claude Code
  ↓
ClaudeBoardHook.exe
  ├─ Hooks：工具调用 / 审批 / 停止事件
  └─ statusLine：模型 / CTX / 费用
  ↓
ClaudeBoardMonitor.exe
  ├─ 图形化状态显示
  ├─ 本地 HTTP 服务 127.0.0.1:8765
  ├─ 外部审批模式
  └─ USART0 串口转发
  ↓
GD32C231C-EVAL
  ├─ LCD：外部 Flash 背景 + 裸 LCD 覆盖层
  ├─ DHT11：空闲状态显示温湿度
  ├─ LED：运行 / 审批 / 错误提示
  └─ PA0 / PA4：同意 / 拒绝
```

从嵌入式角度看，本项目的重点并不是“让小 MCU 强行跑复杂 UI 框架”，而是把任务边界划清楚：

- PC 负责复杂协议、Claude Hooks、JSON、审批等待和图形化配置
- 外部 Flash 负责存储固定 UI 资源
- MCU 负责稳定显示、按键交互、传感器采集和状态灯效

这种分工更符合 GD32C231C8 的资源特点，也更适合后续继续扩展。

## 15. 总结

本文完整记录了 GD32C231C-EVAL 开发板从开箱到实现 Claude 状态监视器的全过程。整个评测遵循"由浅入深、逐层递进"的原则，将一个复杂的嵌入式项目拆解为七个可独立验证的阶段。

### 15.1 技术路线回顾

| 阶段 | 核心内容 | 关键技术 |
|------|----------|----------|
| 阶段一 | GPIO 流水灯 | GPIO 输出、推挽配置、bit mask 控制 |
| 阶段二 | USART 串口链路 | USART0、中断接收、命令协议 |
| 阶段三 | FreeRTOS 移植 | Cortex-M23 port、任务划分、队列通信 |
| 阶段四 | LVGL 屏幕显示 | SPI LCD、flush callback、tick 接入 |
| 阶段五 | DHT11 传感器 | 单总线时序、微秒延时、数据校验 |
| 阶段六 | 外部 Flash UI | GD25Q16、SPI1 模式切换、RLE 压缩 |
| 阶段七 | Claude Code 联动 | Windows 上位机、Hooks、权限审批 |

### 15.2 核心经验总结

**1. 资源受限下的工程策略**

GD32C231C8 的 64 KB Flash 和 12 KB SRAM 决定了不能采用"大 MCU + 大 framebuffer"的方案。本项目的解决思路是：

- 大图和固定素材存入外部 Flash GD25Q16
- 片内 SRAM 仅保留一行解码缓冲（480 字节）
- 动态 UI 采用裸 LCD 矩形和像素数字，不依赖完整 GUI 框架

**2. 多任务架构设计**

FreeRTOS 的引入使程序结构从"主循环轮询"升级为"任务驱动"：

- LED 任务：低频状态刷新
- 串口任务：高优先级命令响应
- GUI 任务：周期性显示更新
- 传感器任务：定时数据采集

各任务通过队列和临界区共享数据，职责清晰，便于维护和扩展。

**3. 外设共用 SPI1 的处理**

LCD 和 GD25Q16 共用 SPI1，但 SPI 模式不同。解决方案是在每次切换设备前调用对应的初始化函数（`lcd_bus_init()` 或 `spi_flash_init()`），通过 `spi_i2s_deinit()` 复位后重新配置。

**4. PC 与嵌入式的分工**

本项目采用三段式架构：

- Claude Code：运行在 PC，负责 AI 推理和 Hooks
- Windows 上位机：运行在 PC，负责协议转换和串口转发
- GD32C231C-EVAL：负责显示、交互和传感器采集

这种分工让 MCU 只承担适合其资源能力的任务，避免了在小芯片上强行实现 HTTPS、JSON 解析等复杂功能。

### 15.3 项目成果

经过七个阶段的迭代，最终实现的功能：

- **LCD 显示**：外部 Flash 背景 + 裸 LCD 覆盖层，显示时间、温湿度、Claude 状态
- **状态动画**：空闲小人（耳机+音符）、工作小人（电脑+代码块+星星）
- **DHT11 温湿度**：实时采集和显示
- **LED 状态指示**：心跳、运行、审批、错误等多种灯效
- **串口协议**：支持 Claude 状态、时钟设置、模型信息等命令
- **权限审批**：PA0 同意、PA4 拒绝，支持预按缓存
- **Windows 上位机**：Claude Code Hooks 集成，图形化状态显示

### 15.5 后续扩展方向

1. **状态图标扩展**：在 `CUI1` 资源包中加入更多状态图标（thinking、error 等）
2. **状态机完善**：使更多 Claude 状态对应不同图标和动画
