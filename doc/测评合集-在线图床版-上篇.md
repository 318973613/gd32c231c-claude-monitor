## 概述

本文档记录 GD32C231C-EVAL 开发板的完整评测过程。评测目标并非简单的 LED 点灯验证，而是按阶段逐步验证开发板的各项能力，最终实现一个桌面版的 Claude 状态监视器：PC 端通过串口将 Claude 状态发送给 GD32C231C，开发板通过 LCD 显示任务状态，同时接入 DHT11 传感器采集温湿度数据，LED 用于指示运行、接收和异常状态。

整个评测过程以教程形式呈现，每个阶段均包含明确的目标、操作步骤、关键代码和运行验证结果，便于读者复现。

### 6.3 LED 控制逻辑

采用 bit mask 方式统一控制 4 个 LED，bit0 至 bit3 分别对应 LED1 至 LED4：

```c
static void led_write_mask(uint8_t mask)
{
    uint32_t i;

    for(i = 0U; i < LED_COUNT; i++) {
        if(0U != (mask & (uint8_t)(1U << i))) {
            gpio_bit_set(led_port[i], led_pin[i]);
        } else {
            gpio_bit_reset(led_port[i], led_pin[i]);
        }
    }
}
```

通过传入不同的 mask 值即可实现任意灯效组合：

- `0x0F`：四灯全亮
- `0x09`：LED1 和 LED4 亮
- `0x06`：LED2 和 LED3 亮

### 6.4 自定义灯效

### 7.3 命令协议设计

为兼容后续 Claude 状态监视器需求，设计了简洁的命令协议。串口助手发送一行命令，开发板收到回车或换行后解析执行。

支持的命令列表：

| 命令 | 开发板响应 | LED 表现 |
| --- | --- | --- |
| `PING` / `P` | `PONG GD32C231C USART OK` | LED3 闪一下 |
| `IDLE` / `I` | `ACK STATE=IDLE` | 清除运行和错误状态 |
| `RUN` / `R` | `ACK STATE=RUNNING` | LED2 亮 |
| `DONE` / `D` | `ACK STATE=DONE` | 四灯快速闪一下 |
| `ERR` / `E` | `ACK STATE=ERROR` | LED4 亮 |
| `HELP` / `H` | 打印命令列表 | LED3 闪一下 |

LED 功能分配：

| LED | 含义 |
| --- | --- |
| LED1 | 系统心跳，表示程序运行中 |
| LED2 | 当前处于 RUN 状态 |
| LED3 | 收到串口命令时闪烁 |
| LED4 | 当前处于 ERROR 状态 |

此分配方案与后续 Claude Monitor 保持一致：`RUN` 对应任务执行中，`ERR` 对应任务出错，`DONE` 对应任务完成，`PING` 用于链路测试。

### 7.4 串口接收逻辑

采用中断方式接收串口数据。原因在于：后续 Claude 状态监视器需同时刷新 LCD、更新 DHT11 数据、控制 LED，若串口采用主循环轮询方式，程序结构将趋于复杂。

初始化时使能 USART0 NVIC 中断及接收中断：

```c
static void serial_irq_config(void)
{
    nvic_irq_enable(USART0_IRQn, 0U);
    usart_interrupt_enable(EVAL_COM, USART_INT_RBNE);
}
```

中断服务函数 `USART0_IRQHandler()` 仅执行轻量操作：读取数据寄存器、字符转大写、遇回车换行置位 `rx_ready` 标志：

```c
void USART0_IRQHandler(void)
{
    uint8_t ch;

    if(RESET != usart_interrupt_flag_get(EVAL_COM, USART_INT_FLAG_RBNE)) {
        ch = (uint8_t)usart_data_receive(EVAL_COM);

        if((ch == '\r') || (ch == '\n')) {
            if(rx_len > 0U) {
                rx_line[rx_len] = '\0';
                rx_ready_len = rx_len;
                rx_ready = 1U;
            }
        }
    }
}
```

主循环保留，但不再轮询串口硬件，仅处理状态机：

```c
while(1) {
    serial_task();
    led_task();
    delay_1ms(LOOP_TICK_MS);
}
```

`serial_task()` 检测到 `rx_ready` 后复制命令并解析。此设计确保 USART 接收不丢字节，命令处理不阻塞中断。

为便于调试，添加了接收调试输出。正常发送 `PING` 时应看到：

```text
RX[4]: PING
HEX: 50 49 4E 47
PONG GD32C231C USART OK
```

如看到 `RX[10]: PPPPPPPPPP`、`HEX: 50 50 50 50 ...`，说明芯片实际收到的就是一串 `P`，需检查：

- 串口助手是否开启了定时发送
- 发送框是否只选中了 `P`
- USB-TTL 的 TX 是否正确接到开发板 `PA10/RX`
- 双方 GND 是否已共地

## 8. 阶段三：FreeRTOS 移植与任务化串口链路

第二阶段已验证 USART0 命令链路功能正常，但其本质仍为裸机状态机实现。在点灯、串口收发等简单场景下，裸机方案可行；但当后续需同时处理 LCD 刷新、DHT11 采集、串口协议解析及异常状态提示时，主循环逻辑将趋于复杂，维护难度显著增加。

因此本阶段进行 FreeRTOS 移植。移植目标并非简单运行现成例程，而是在 `project` 目录下新建独立工程，将第二阶段的串口状态链路改造为 RTOS 任务模型。

### 8.3 FreeRTOSConfig.h 关键配置

`FreeRTOSConfig.h` 是 FreeRTOS 移植的核心配置文件。本工程关键配置如下：

```c
#define configCPU_CLOCK_HZ                         (SystemCoreClock)
#define configTICK_RATE_HZ                         ((TickType_t)1000U)
#define configMAX_PRIORITIES                       5U
#define configMINIMAL_STACK_SIZE                   128U
#define configTOTAL_HEAP_SIZE                      (8U * 1024U)

#define configENABLE_FPU                           0
#define configENABLE_MPU                           0
#define configENABLE_TRUSTZONE                     0
#define configRUN_FREERTOS_SECURE_ONLY             0

#define configPRIO_BITS                            __NVIC_PRIO_BITS
#define configLIBRARY_LOWEST_INTERRUPT_PRIORITY    3U
#define configLIBRARY_MAX_SYSCALL_INTERRUPT_PRIORITY 2U
#define configKERNEL_INTERRUPT_PRIORITY            (configLIBRARY_LOWEST_INTERRUPT_PRIORITY << (8U - configPRIO_BITS))
#define configMAX_SYSCALL_INTERRUPT_PRIORITY       (configLIBRARY_MAX_SYSCALL_INTERRUPT_PRIORITY << (8U - configPRIO_BITS))
```

配置说明：

- `configTICK_RATE_HZ` 设置为 `1000U`，即 FreeRTOS 系统节拍周期为 1 ms
- GD32C231C NVIC 优先级位数为 2 位，中断优先级配置必须按 `__NVIC_PRIO_BITS` 进行移位

### 8.4 FreeRTOS 心跳定时器

FreeRTOS 系统节拍使用 Cortex-M23 内核自带的 `SysTick` 定时器，而非额外的外设定时器（如 `TIMER15` 或 `TIMER16`）。

当前工程的异常处理函数分配：

| 异常处理函数 | 来源 |
| --- | --- |
| `SysTick_Handler()` | FreeRTOS `port.c` |
| `SVC_Handler()` | FreeRTOS `portasm.c` |
| `PendSV_Handler()` | FreeRTOS `portasm.c` |

`gd32c2x1_it.c` 中不再定义上述三个异常处理函数。FreeRTOS 调度入口完全由官方 Cortex-M23 port 接管：

- `SysTick`：产生 RTOS tick
- `PendSV`：执行上下文切换
- `SVC`：启动首个任务

链接 MAP 可验证异常向量已正确绑定：

```text

startup_gd32c2x1.o(RESET) refers to portasm.o(.text.PendSV_Handler)
startup_gd32c2x1.o(RESET) refers to port.o(.text.SysTick_Handler)
```

### 8.5 任务划分

本阶段将裸机串口状态链路拆分为三个 FreeRTOS 任务：

| 任务 | 优先级 | 功能描述 |
| --- | --- | --- |
| `LED` task | `tskIDLE_PRIORITY + 1` | 刷新 LED 状态（心跳、RUN、RX、ERROR） |
| `SERIAL` task | `tskIDLE_PRIORITY + 2` | 从串口队列取数据，拼接并解析命令 |
| `MONITOR` task | `tskIDLE_PRIORITY + 1` | 周期打印 RTOS tick、队列占用及剩余堆空间 |

任务创建代码：

```c
xTaskCreate(led_task, “led”, LED_TASK_STACK_WORDS, NULL, LED_TASK_PRIORITY, NULL);
xTaskCreate(serial_task, “serial”, SERIAL_TASK_STACK_WORDS, NULL, SERIAL_TASK_PRIORITY, NULL);
xTaskCreate(monitor_task, “monitor”, MONITOR_TASK_STACK_WORDS, NULL, MONITOR_TASK_PRIORITY, NULL);

serial_irq_config();
vTaskStartScheduler();
```

`SERIAL` 任务优先级高于其他两个任务，原因在于串口命令属于外部输入事件，响应优先级应高于常规状态刷新。LED 和 Monitor 任务为低频状态任务，无需过高优先级。

### 8.6 USART 中断与队列机制

串口接收仍采用 USART0 中断方式，但中断处理逻辑调整为：将接收到的每个字节投递至 FreeRTOS 队列，而非直接在中断中拼接命令。

```c
void USART0_IRQHandler(void)
{
    uint8_t ch;
    BaseType_t higher_priority_task_woken = pdFALSE;

    if(RESET != usart_interrupt_flag_get(EVAL_COM, USART_INT_FLAG_RBNE)) {
        ch = (uint8_t)usart_data_receive(EVAL_COM);
        if(NULL != rx_queue) {
            (void)xQueueSendFromISR(rx_queue, &ch, &higher_priority_task_woken);
            portYIELD_FROM_ISR(higher_priority_task_woken);
        }
    }
}
```

此设计优势：

- 中断仅执行”接收字节 + 投递队列”操作，执行时间短
- 命令拼接和字符串解析在 `SERIAL` 任务中完成
- 后续可通过队列或任务通知扩展 LCD 刷新、DHT11 状态更新等功能

串口中断优先级配置：

```c
nvic_irq_enable(USART0_IRQn, configLIBRARY_MAX_SYSCALL_INTERRUPT_PRIORITY);
```

由于中断中调用了 `xQueueSendFromISR()`，中断优先级不得高于 `configMAX_SYSCALL_INTERRUPT_PRIORITY`，以确保符合 FreeRTOS 临界区规则。

### 8.7 LED 状态设计

LED 功能分配沿用第二阶段设计，便于与后续 Claude Monitor 对接：

| LED | 功能描述 |
| --- | --- |
| LED1 | FreeRTOS 任务心跳，由 `LED` 任务基于 `xTaskGetTickCount()` 翻转 |
| LED2 | `RUN` 状态指示 |
| LED3 | 串口 RX 脉冲指示 |
| LED4 | `ERROR` 状态指示 |

> **说明**：LED1 表示调度器和 LED 任务正常运行，与 FreeRTOS 内核 tick（由 `SysTick_Handler()` 提供）是两个不同概念。

LED 任务逻辑：

```c
last_wake = xTaskGetTickCount();
last_heartbeat = last_wake;

while(1) {
    now = xTaskGetTickCount();
    if((now - last_heartbeat) >= pdMS_TO_TICKS(HEARTBEAT_PERIOD_MS)) {
        last_heartbeat = now;
        heartbeat_on = (uint8_t)!heartbeat_on;
    }

    led_write(0U, heartbeat_on);
    led_write(1U, state_running);
    led_write(2U, (uint8_t)(now < rx_pulse_until));
    led_write(3U, state_error);

    vTaskDelayUntil(&last_wake, pdMS_TO_TICKS(50U));
}
```
