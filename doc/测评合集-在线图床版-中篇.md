## 9. 阶段四：FreeRTOS + LVGL 屏幕显示移植

本阶段在 FreeRTOS 串口链路基础上，进一步实现 LCD 屏幕显示功能。未直接基于官方 SPI LCD 例程修改，而是在 `project` 目录下新建独立工程，完整集成 LCD 驱动、LVGL 显示框架、FreeRTOS tick 及内存裁剪方案。

本阶段目标明确：暂不涉及串口和 LED 功能，仅验证 FreeRTOS 任务环境下 LVGL 能否稳定驱动 SPI LCD 显示。此阶段验证通过后，后续可逐步接入 DHT11、串口状态协议及 Claude 状态监视器，整体架构保持清晰。

### 9.2 LVGL 版本选择

本工程选用 LVGL `v8.4.0`，未采用 LVGL v9。选择依据如下：

- 当前芯片为 `GD32C231C8`，资源有限（64 KB Flash / 12 KB SRAM）
- LVGL v8 配置项成熟，裁剪路径清晰
- 更适合小 Flash、小 SRAM 的 Cortex-M23 工程

本阶段目标并非构建复杂 UI，而是建立稳定的显示链路：

```text
FreeRTOS GUI task
    ↓
LVGL object / draw buffer
    ↓
LVGL flush callback
    ↓
lcd_flush_area()
    ↓
SPI1 -> LCD GRAM
```

此链路验证通过后，后续显示任务状态、温湿度数据或 Claude 状态面板，均可在 LVGL 对象层扩展。

### 9.3 LCD 硬件连接

开发板 LCD 使用 SPI 接口，控制器兼容 ILI9341 规格。本工程沿用官方 LCD 初始化序列，主要引脚分配如下：

| 功能 | 引脚 |
| --- | --- |
| SPI1_SCK | PB13 |
| SPI1_MISO | PB14 |
| SPI1_MOSI | PB15 |
| LCD_CS | PA8 |
| LCD_DC / RS | PC7 |
| LCD_RST | PC6 |

屏幕分辨率配置（竖屏模式）：

```c
#define LCD_HOR_RES              240U
#define LCD_VER_RES              320U
```

### 9.4 FreeRTOS 任务设计

本阶段仅创建一个 GUI 任务，优先验证显示链路：

```c
#define GUI_TASK_STACK_WORDS     768U
#define GUI_TASK_PRIORITY        (tskIDLE_PRIORITY + 2U)

int main(void)
{
    if(pdPASS != xTaskCreate(gui_task, “gui”, GUI_TASK_STACK_WORDS, NULL, GUI_TASK_PRIORITY, NULL)) {
        while(1) {
        }
    }

    vTaskStartScheduler();

    while(1) {
    }
}
```

`gui_task()` 工作流程：

```c
static void gui_task(void *argument)
{
    (void)argument;

    lcd_init();
    lcd_clear(BLACK);
    lv_init();
    lvgl_port_init();
    create_demo_screen();

    while(1) {
        lv_timer_handler();
        vTaskDelay(pdMS_TO_TICKS(5U));
    }
}
```

> **说明**：LVGL 运行于 FreeRTOS GUI 任务中，`lv_timer_handler()` 由任务周期调用，任务通过 `vTaskDelay()` 让出 CPU。后续新增 DHT11 采集任务或串口协议任务时，无需重构当前架构。

### 9.5 LVGL Tick 接入 FreeRTOS

LVGL 需要毫秒级 tick 用于定时器、动画和刷新周期判断。本工程启用 `LV_TICK_CUSTOM`，使 LVGL 直接使用 FreeRTOS tick：

```c
#define LV_TICK_CUSTOM 1
#define LV_TICK_CUSTOM_INCLUDE “lv_port_tick.h”
#define LV_TICK_CUSTOM_SYS_TIME_EXPR (xTaskGetTickCount())
```

`lv_port_tick.h` 确保 FreeRTOS 头文件包含顺序正确：

```c
#ifndef LV_PORT_TICK_H
#define LV_PORT_TICK_H

#include “FreeRTOS.h”
#include “task.h”

#endif /* LV_PORT_TICK_H */
```

> **注意**：若让 LVGL 直接包含 `task.h`，FreeRTOS 将报编译错误，因为 `task.h` 必须在 `FreeRTOS.h` 之后包含。通过 `lv_port_tick.h` 包装头可解决此问题。

### 9.6 LVGL 显示驱动注册

LVGL 通过注册 `flush_cb` 回调实现底层屏幕刷新。本工程使用 2 行显示缓冲，在保证正常刷新的同时控制 SRAM 占用：

```c
#define LVGL_BUFFER_LINES        2U

static lv_disp_draw_buf_t draw_buf;
static lv_color_t lvgl_buf[LCD_HOR_RES * LVGL_BUFFER_LINES];
```

显示驱动注册：

```c
static void lvgl_port_init(void)
{
    static lv_disp_drv_t disp_drv;

    lv_disp_draw_buf_init(&draw_buf, lvgl_buf, NULL, LCD_HOR_RES * LVGL_BUFFER_LINES);

    lv_disp_drv_init(&disp_drv);
    disp_drv.hor_res = LCD_HOR_RES;
    disp_drv.ver_res = LCD_VER_RES;
    disp_drv.flush_cb = lvgl_flush_cb;
    disp_drv.draw_buf = &draw_buf;
    (void)lv_disp_drv_register(&disp_drv);
}
```

刷新回调将 LVGL 输出的 RGB565 像素数据写入 LCD：

```c
static void lvgl_flush_cb(lv_disp_drv_t *disp_drv, const lv_area_t *area, lv_color_t *color_p)
{
    if((area->x2 < 0) || (area->y2 < 0) ||
       (area->x1 >= (lv_coord_t)LCD_HOR_RES) || (area->y1 >= (lv_coord_t)LCD_VER_RES)) {
        lv_disp_flush_ready(disp_drv);
        return;
    }

    lcd_flush_area((uint16_t)area->x1,
                   (uint16_t)area->y1,
                   (uint16_t)area->x2,
                   (uint16_t)area->y2,
                   (const uint16_t *)color_p);

    lv_disp_flush_ready(disp_drv);
}
```

### 9.7 LCD 区域刷新函数

官方 LCD 例程提供点、线、色块等裸机显示接口。为适配 LVGL，新增区域刷新函数 `lcd_flush_area()`：

```c
void lcd_flush_area(uint16_t x_start, uint16_t y_start, uint16_t x_end, uint16_t y_end, const uint16_t *color_p)
{
    uint32_t count;
    uint16_t color;

    count = ((uint32_t)x_end - x_start + 1U) * ((uint32_t)y_end - y_start + 1U);
    lcd_set_region(x_start, y_start, x_end, y_end);
    LCD_RS_SET;
    LCD_CS_CLR;

    while(count > 0U) {
        color = *color_p++;
        spi_write_byte(SPI1, (uint8_t)(color >> 8));
        spi_write_byte(SPI1, (uint8_t)color);
        count--;
    }

    LCD_CS_SET;
}
```

此函数是 LVGL 移植的关键点。LVGL 仅告知”需要刷新的区域及像素数据”，具体的 LCD 地址窗口设置和 SPI RGB565 写入由 BSP 层完成。

### 9.9 显示问题修复

首次下载后屏幕出现蓝色横纹，表明 LCD 初始化和 SPI 通信已通，但显示内容不完整。解决方案如下：

**问题一：蓝色横纹**

LCD 初始化后先执行 `lcd_clear(BLACK)`，清除 GRAM 中的旧数据。LVGL 根屏幕背景显式设置不透明度：

```c
screen = lv_scr_act();
lv_obj_clear_flag(screen, LV_OBJ_FLAG_SCROLLABLE);
lv_obj_set_style_bg_color(screen, lv_color_hex(0x102030), 0);
lv_obj_set_style_bg_opa(screen, LV_OPA_COVER, 0);
```

**问题二：显示方向**

根据实际安装方向，将 LCD 显示旋转 180 度。通过修改 LCD 控制器 MADCTL 寄存器（`0x36`）实现：

```c
#ifdef H_VIEW
#define LCD_MADCTL          (uint8_t)0x28
#else
#define LCD_MADCTL          (uint8_t)0x88
#endif
```

初始化时写入：

```c
lcd_write_index(0x36);
lcd_write_data(LCD_MADCTL);
```

> **说明**：此方法直接修改 LCD 控制器扫描方向，而非修改 LVGL 坐标系，适用于固定安装方向的屏幕。

### 10.2 DHT11 引脚规划

DHT11 DATA 引脚规划为 `PB0`：

| DHT11 引脚 | GD32C231C-EVAL 连接 |
| --- | --- |
| VCC | 3V3 |
| GND | GND |
| DATA | PB0 |

![DHT11 接线实物图](https://raw.githubusercontent.com/318973613/gd32c231c-claude-monitor/master/doc/a053b7c2858af7cb1dee0f85003a6e07.jpg)

引脚选择依据：

- LCD 已占用 SPI1 相关引脚（PB13/PB14/PB15）及控制引脚（PA8/PC7/PC6）
- 后续 Claude 状态监视器需预留 USART0（PA9/PA10）
- PB0 不与上述外设冲突

> **注意**：DHT11 DATA 需上拉至 3.3V。常见三针或四针 DHT11 模块一般自带上拉电阻；裸 DHT11 传感器需在 DATA 与 3V3 之间外接 `4.7kΩ ~ 10kΩ` 上拉电阻。

### 10.3 任务结构设计

本阶段保留 GUI 任务，新增 DHT11 采集任务：

| 任务 | 优先级 | 周期 | 功能 |
| --- | --- | --- | --- |
| `gui` | `tskIDLE_PRIORITY + 2` | 5 ms（LVGL handler），500 ms（显示刷新） | LCD + LVGL 显示 |
| `dht11` | `tskIDLE_PRIORITY + 1` | 2 s | DHT11 温湿度采集 |

任务创建代码：

```c
#define GUI_TASK_STACK_WORDS     768U
#define GUI_TASK_PRIORITY        (tskIDLE_PRIORITY + 2U)
#define DHT_TASK_STACK_WORDS     160U
#define DHT_TASK_PRIORITY        (tskIDLE_PRIORITY + 1U)

    while(1) {
    }
}

if(pdPASS != xTaskCreate(dht11_task, “dht11”, DHT_TASK_STACK_WORDS, NULL, DHT_TASK_PRIORITY, NULL)) {
    while(1) {
    }
}

vTaskStartScheduler();
```

> **设计说明**：DHT11 采用单总线时序，读取需微秒级精确采样。若将其置于 GUI 任务中，会将显示刷新逻辑与传感器时序耦合。独立任务设计使 GUI 仅需读取最新状态并刷新 label，结构更清晰。

### 10.5 单总线时序

DHT11 通信流程：

1. MCU 拉低 DATA 至少 18 ms（启动信号）
2. MCU 释放 DATA
3. DHT11 响应低电平约 80 μs，再拉高约 80 μs
4. DHT11 输出 40 bit 数据
5. 每 bit 先低电平约 50 μs，高电平宽度决定 0 或 1

启动信号使用 FreeRTOS 延时：

```c
dht11_drive_low();
vTaskDelay(pdMS_TO_TICKS(DHT11_START_MS));
```

数据采样阶段使用短临界区保护：

```c
taskENTER_CRITICAL();

dht11_release_line();
dht11_delay_us(DHT11_RELEASE_US);

/* response and 40-bit sample */

taskEXIT_CRITICAL();
```

> **说明**：DHT11 的 0/1 判断依赖 26 μs 和 70 μs 高电平差异。若采样过程被 SysTick 或其他中断打断，可能导致误判。

### 10.6 数据采样优化

初始方案采用”统计高电平持续时间”判断 0/1，理论上可行，但在 Cortex-M23 + 编译优化后，空循环延时不稳定，易将 70 μs 高电平误判为短脉冲，导致 40 bit 全部读为 0。

DHT11 校验规则：

```text
checksum = 湿度整数 + 湿度小数 + 温度整数 + 温度小数
```

若读取到 `00 00 00 00 00`，校验和恰好为 0，旧逻辑误判为 `OK`，屏幕显示 `0.0 C / 0.0 %`。

优化方案：每个 bit 高电平开始后延时约 40 μs 再采样：

```c
dht11_delay_us(DHT11_BIT_SAMPLE_US);

raw[bit / 8U] <<= 1U;
if(0U != dht11_read_pin()) {
    raw[bit / 8U] |= 1U;
}
```

判断逻辑：

- 40 μs 后 DATA 已变低 → 短高电平 → bit = 0
- 40 μs 后 DATA 仍为高 → 长高电平 → bit = 1

新增全 0 数据保护：

```c
if((0U == raw[0]) && (0U == raw[1]) && (0U == raw[2]) && (0U == raw[3]) && (0U == raw[4])) {
    return DHT11_ERROR_ZERO_DATA;
}
```

屏幕显示 `ZERO` 而非误报有效温湿度，便于排查时序或接线问题。

### 10.7 微秒延时实现

普通空循环延时受编译优化影响明显。本工程采用基于 `SysTick->VAL` 的微秒延时，利用 FreeRTOS 已使用的 SysTick 定时器当前递减计数值估算经过的 CPU tick：

```c
static void dht11_delay_us(uint32_t us)
{
    uint32_t ticks_per_us;
    uint32_t target_ticks;
    uint32_t elapsed_ticks = 0U;
    uint32_t reload;
    uint32_t last;
    uint32_t now;

    ticks_per_us = SystemCoreClock / 1000000U;
    if(0U == ticks_per_us) {
        ticks_per_us = 1U;
    }

    target_ticks = us * ticks_per_us;
    reload = SysTick->LOAD;
    last = SysTick->VAL;

    while(elapsed_ticks < target_ticks) {
        now = SysTick->VAL;
        if(last >= now) {
            elapsed_ticks += last - now;
        } else {
            elapsed_ticks += last + (reload - now) + 1U;
        }
        last = now;
    }
}
```

> **技术要点**：DHT11 单总线时序对延时精度敏感。在 RTOS 环境下，需明确哪些位置可让出 CPU，哪些位置必须短时间保持时序稳定。

### 10.8 DHT11 采集任务

DHT11 任务每 2 秒采集一次，结果写入共享状态：

```c
static void dht11_task(void *argument)
{
    dht11_data_t data;
    dht11_status_t status;

    (void)argument;

    dht11_init();
    vTaskDelay(pdMS_TO_TICKS(1500U));

    while(1) {
        status = dht11_read(&data);

        taskENTER_CRITICAL();
        sensor_state.status = status;
        sensor_state.sample_count++;
        if(DHT11_OK == status) {
            sensor_state.data = data;
        } else {
            sensor_state.error_count++;
        }
        taskEXIT_CRITICAL();

        vTaskDelay(pdMS_TO_TICKS(DHT_SAMPLE_PERIOD_MS));
    }
}
```

`sample_count` 和 `error_count` 用于现场调试，可快速判断传感器是偶发错误还是持续错误。

## 12. 阶段六：Claude UI 资源外部 Flash 方案

完成 `FreeRTOS + LVGL + DHT11` 集成后，开发板已具备屏幕显示、传感器采集和串口通信能力。但在实现 Claude 风格界面时，面临一个现实问题：GD32C231C8 片内资源有限（64 KB Flash / 12 KB SRAM），若将图片、大字体、复杂 UI 控件全部置于内部 Flash 或 SRAM，工程资源将迅速耗尽。

本阶段采用更接近实际产品的方案：将固定 UI 资源预烧至板载外部 Flash，应用程序启动后从外部 Flash 读取资源显示。主程序仅负责逻辑处理、传感器数据采集、串口通信及动态数据更新，图片和界面素材交由外部 Flash 存储。

![外部 Flash 资源方案架构](https://raw.githubusercontent.com/318973613/gd32c231c-claude-monitor/master/doc/9fe9fc4ee24ee0e8a4e8d49a7ef7c392.jpeg)

### 12.1 方案选择依据

开发板片内 SRAM 有限，LVGL 阶段已显现资源压力。第 05 阶段加入 DHT11 后，静态 SRAM 占用已接近 11 KB。后续若继续增加串口、时钟、状态界面、图片或大字体，片内资源将难以承受。

初期尝试使用 LVGL label 构建 Claude 风格界面，虽可显示但效果有限，且对象数量增加后 LVGL heap 和 FreeRTOS heap 均趋于紧张。因此调整思路：固定不变的背景图、装饰图、文案资源不应占用 MCU 片内 Flash，更不应长期占用 SRAM。

本阶段最终采用的架构：

```text
PC 生成 UI 资源包
        ↓
资源烧录工程 07 写入 GD25Q16
        ↓
应用工程 08 启动后读取 GD25Q16
        ↓
LCD 显示背景 + 动态叠加时间 / DHT11 / 串口状态
```

方案优势：

- UI 图片容量可超过片内 Flash 限制
- 应用固件无需因更换背景图而频繁修改代码
- SRAM 仅需一行解码缓冲，无需整屏 framebuffer
- 后续接入 Claude 或其他上位机协议时，应用逻辑更清晰

### 12.7 SPI1 总线切换处理

Flash 和 LCD 共用 SPI1，但 SPI 模式不同，这是本阶段的关键技术难点。解决方案是在 LCD 驱动中增加总线初始化接口：

```c
void lcd_bus_init(void)
{
    spi1_init();
}
```

`spi1_init()` 和 `spi_flash_init()` 中均先复位 SPI1：

```c
spi_i2s_deinit(SPI1);
```

每次切换设备时，SPI1 按当前设备需要的模式重新配置。

背景图绘制流程：

```text
切到 Flash 模式
读取一段 RLE 数据
解码到一行 line_buffer
切回 LCD 模式
将该行刷到 LCD
继续下一行
```

此方案虽非最快，但对 12 KB SRAM 的芯片稳定可靠。未申请整屏 framebuffer，仅保留一行 240 像素的 RGB565 缓冲：

```c
static uint16_t line_buffer[240];
```

占用约 480 字节 SRAM。

### 12.12 本阶段总结

本阶段验证的核心结论：GD32C231C（64 KB Flash / 12 KB SRAM）不能采用”大 MCU + 大 framebuffer”的 UI 方案。可行方式是将固定资源与动态覆盖层分离。

当前架构：

```text
资源生成脚本 generate_ui_assets.py
  ↓
生成 ui_assets.bin / ui_assets.c / 预览图
  ↓
第 07 工程将 CUI1 资源包写入 GD25Q16
  ↓
第 08 工程启动时从 GD25Q16 读取背景
  ↓
LCD 显示静态背景
  ↓
裸 LCD 小区域刷新时钟 / DHT11 / 计数器 / 状态头像 / 动画
```

工程职责划分：

- 第 07 工程：资源烧录器，负责将 UI 资源包可靠写入 GD25Q16 并读回校验
- 第 08 工程：实际应用，负责 FreeRTOS、DHT11、USART0 时钟、LED 灯效、LCD 显示及运行时小区域刷新

当前第 08 工程虽保留 LVGL 源码参与编译，但实际 UI 路径未调用 `lv_init()`、`lv_timer_handler()`、`lv_obj_*()` 或 `lv_label_*()`。此为有意保守处理：先稳定外部 Flash 资源、SPI1 模式切换、DHT11、串口和 LCD 刷新，再考虑是否重新引入 LVGL 部分能力。

关键经验：

1. 大图和固定素材存入 GD25Q16，片内 SRAM 仅保留一行 `line_buffer[240]` 和少量覆盖层缓冲
2. LCD 和 GD25Q16 共用 SPI1 时，读 Flash 前须调用 `spi_flash_init()`，画 LCD 前须调用 `lcd_bus_init()`，因两者 SPI 模式不同
3. 动态 UI 不必依赖完整 GUI 框架，240x320 小屏使用像素数字、矩形卡片和小区域动画更易控制内存和刷新范围
4. 动画应避免整块刷新，仅刷新变化小块可保持连贯性且不拖慢主循环

后续扩展方向：

1. 扩展 `CUI1` 资源包，加入更多状态图标或背景（idle、working、thinking、error）
2. 完善状态机，使未连接、等待命令、DHT11 错误、串口收到命令等状态对应不同图标
3. UI 和底层链路稳定后接入 Claude 通信协议，通信状态走 USART 或其他通道，固定视觉资源继续存外部 Flash，动态状态由第 08 工程小区域刷新
