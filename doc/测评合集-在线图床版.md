# 【兆易创新GD32C231C评测】从开箱到 Claude 状态监视器：LCD、串口与 DHT11 综合实战
![alt text](https://raw.githubusercontent.com/318973613/gd32c231c-claude-monitor/master/doc/94197ba3b4a2ef4b31223dee7c86be16.png)
## 概述

本文档记录 GD32C231C-EVAL 开发板的完整评测过程。评测目标并非简单的 LED 点灯验证，而是按阶段逐步验证开发板的各项能力，最终实现一个桌面版的 Claude 状态监视器：PC 端通过串口将 Claude 状态发送给 GD32C231C，开发板通过 LCD 显示任务状态，同时接入 DHT11 传感器采集温湿度数据，LED 用于指示运行、接收和异常状态。

整个评测过程以教程形式呈现，每个阶段均包含明确的目标、操作步骤、关键代码和运行验证结果，便于读者复现。

### 6.3 LED 控制逻辑

采用 bit mask 方式统一控制 4 个 LED，bit0 至 bit3 分别对应 LED1 至 LED4：

```c
static void led_write_mask(uint8_t mask)
{
    uint32_t i;

    for(i = 0U; i < LED_COUNT; i++) {
        if(0U != (mask & (uint8_t)(1U << i))) {
            gpio_bit_set(led_port[i], led_pin[i]);
        } else {
            gpio_bit_reset(led_port[i], led_pin[i]);
        }
    }
}
```

通过传入不同的 mask 值即可实现任意灯效组合：

- `0x0F`：四灯全亮
- `0x09`：LED1 和 LED4 亮
- `0x06`：LED2 和 LED3 亮

### 7.3 命令协议设计

为兼容后续 Claude 状态监视器需求，设计了简洁的命令协议。串口助手发送一行命令，开发板收到回车或换行后解析执行。

支持的命令列表：

| 命令 | 开发板响应 | LED 表现 |
| --- | --- | --- |
| `PING` / `P` | `PONG GD32C231C USART OK` | LED3 闪一下 |
| `IDLE` / `I` | `ACK STATE=IDLE` | 清除运行和错误状态 |
| `RUN` / `R` | `ACK STATE=RUNNING` | LED2 亮 |
| `DONE` / `D` | `ACK STATE=DONE` | 四灯快速闪一下 |
| `ERR` / `E` | `ACK STATE=ERROR` | LED4 亮 |
| `HELP` / `H` | 打印命令列表 | LED3 闪一下 |

LED 功能分配：

| LED | 含义 |
| --- | --- |
| LED1 | 系统心跳，表示程序运行中 |
| LED2 | 当前处于 RUN 状态 |
| LED3 | 收到串口命令时闪烁 |
| LED4 | 当前处于 ERROR 状态 |

此分配方案与后续 Claude Monitor 保持一致：`RUN` 对应任务执行中，`ERR` 对应任务出错，`DONE` 对应任务完成，`PING` 用于链路测试。

### 7.4 串口接收逻辑

采用中断方式接收串口数据。原因在于：后续 Claude 状态监视器需同时刷新 LCD、更新 DHT11 数据、控制 LED，若串口采用主循环轮询方式，程序结构将趋于复杂。

初始化时使能 USART0 NVIC 中断及接收中断：

```c
static void serial_irq_config(void)
{
    nvic_irq_enable(USART0_IRQn, 0U);
    usart_interrupt_enable(EVAL_COM, USART_INT_RBNE);
}
```

中断服务函数 `USART0_IRQHandler()` 仅执行轻量操作：读取数据寄存器、字符转大写、遇回车换行置位 `rx_ready` 标志：

```c
void USART0_IRQHandler(void)
{
    uint8_t ch;

    if(RESET != usart_interrupt_flag_get(EVAL_COM, USART_INT_FLAG_RBNE)) {
        ch = (uint8_t)usart_data_receive(EVAL_COM);

        if((ch == '\r') || (ch == '\n')) {
            if(rx_len > 0U) {
                rx_line[rx_len] = '\0';
                rx_ready_len = rx_len;
                rx_ready = 1U;
            }
        }
    }
}
```

主循环保留，但不再轮询串口硬件，仅处理状态机：

```c
while(1) {
    serial_task();
    led_task();
    delay_1ms(LOOP_TICK_MS);
}
```

`serial_task()` 检测到 `rx_ready` 后复制命令并解析。此设计确保 USART 接收不丢字节，命令处理不阻塞中断。

为便于调试，添加了接收调试输出。正常发送 `PING` 时应看到：

```text
RX[4]: PING
HEX: 50 49 4E 47
PONG GD32C231C USART OK
```

如看到 `RX[10]: PPPPPPPPPP`、`HEX: 50 50 50 50 ...`，说明芯片实际收到的就是一串 `P`，需检查：

- 串口助手是否开启了定时发送
- 发送框是否只选中了 `P`
- USB-TTL 的 TX 是否正确接到开发板 `PA10/RX`
- 双方 GND 是否已共地

第二阶段已验证 USART0 命令链路功能正常，但其本质仍为裸机状态机实现。在点灯、串口收发等简单场景下，裸机方案可行；但当后续需同时处理 LCD 刷新、DHT11 采集、串口协议解析及异常状态提示时，主循环逻辑将趋于复杂，维护难度显著增加。

### 8.4 FreeRTOS 心跳定时器

FreeRTOS 系统节拍使用 Cortex-M23 内核自带的 `SysTick` 定时器，而非额外的外设定时器（如 `TIMER15` 或 `TIMER16`）。

| 异常处理函数 | 来源 |
| --- | --- |
| `SysTick_Handler()` | FreeRTOS `port.c` |
| `SVC_Handler()` | FreeRTOS `portasm.c` |
| `PendSV_Handler()` | FreeRTOS `portasm.c` |

`gd32c2x1_it.c` 中不再定义上述三个异常处理函数。FreeRTOS 调度入口完全由官方 Cortex-M23 port 接管：

- `SysTick`：产生 RTOS tick
- `PendSV`：执行上下文切换
- `SVC`：启动首个任务

### 8.5 任务划分

本阶段将裸机串口状态链路拆分为三个 FreeRTOS 任务：

| 任务 | 优先级 | 功能描述 |
| --- | --- | --- |
| `LED` task | `tskIDLE_PRIORITY + 1` | 刷新 LED 状态（心跳、RUN、RX、ERROR） |
| `SERIAL` task | `tskIDLE_PRIORITY + 2` | 从串口队列取数据，拼接并解析命令 |
| `MONITOR` task | `tskIDLE_PRIORITY + 1` | 周期打印 RTOS tick、队列占用及剩余堆空间 |

任务创建代码：

```c
xTaskCreate(led_task, “led”, LED_TASK_STACK_WORDS, NULL, LED_TASK_PRIORITY, NULL);
xTaskCreate(serial_task, “serial”, SERIAL_TASK_STACK_WORDS, NULL, SERIAL_TASK_PRIORITY, NULL);
xTaskCreate(monitor_task, “monitor”, MONITOR_TASK_STACK_WORDS, NULL, MONITOR_TASK_PRIORITY, NULL);

serial_irq_config();
vTaskStartScheduler();
```

`SERIAL` 任务优先级高于其他两个任务，原因在于串口命令属于外部输入事件，响应优先级应高于常规状态刷新。LED 和 Monitor 任务为低频状态任务，无需过高优先级。

### 8.6 USART 中断与队列机制

串口接收仍采用 USART0 中断方式，但中断处理逻辑调整为：将接收到的每个字节投递至 FreeRTOS 队列，而非直接在中断中拼接命令。

```c
void USART0_IRQHandler(void)
{
    uint8_t ch;
    BaseType_t higher_priority_task_woken = pdFALSE;

    if(RESET != usart_interrupt_flag_get(EVAL_COM, USART_INT_FLAG_RBNE)) {
        ch = (uint8_t)usart_data_receive(EVAL_COM);
        if(NULL != rx_queue) {
            (void)xQueueSendFromISR(rx_queue, &ch, &higher_priority_task_woken);
            portYIELD_FROM_ISR(higher_priority_task_woken);
        }
    }
}
```

此设计优势：

- 中断仅执行”接收字节 + 投递队列”操作，执行时间短
- 命令拼接和字符串解析在 `SERIAL` 任务中完成
- 后续可通过队列或任务通知扩展 LCD 刷新、DHT11 状态更新等功能

串口中断优先级配置：

```c
nvic_irq_enable(USART0_IRQn, configLIBRARY_MAX_SYSCALL_INTERRUPT_PRIORITY);
```

由于中断中调用了 `xQueueSendFromISR()`，中断优先级不得高于 `configMAX_SYSCALL_INTERRUPT_PRIORITY`，以确保符合 FreeRTOS 临界区规则。

### 8.7 LED 状态设计

LED 功能分配沿用第二阶段设计，便于与后续 Claude Monitor 对接：

| LED | 功能描述 |
| --- | --- |
| LED1 | FreeRTOS 任务心跳，由 `LED` 任务基于 `xTaskGetTickCount()` 翻转 |
| LED2 | `RUN` 状态指示 |
| LED3 | 串口 RX 脉冲指示 |
| LED4 | `ERROR` 状态指示 |

> **说明**：LED1 表示调度器和 LED 任务正常运行，与 FreeRTOS 内核 tick（由 `SysTick_Handler()` 提供）是两个不同概念。

LED 任务逻辑：

```c
last_wake = xTaskGetTickCount();
last_heartbeat = last_wake;

while(1) {
    now = xTaskGetTickCount();
    if((now - last_heartbeat) >= pdMS_TO_TICKS(HEARTBEAT_PERIOD_MS)) {
        last_heartbeat = now;
        heartbeat_on = (uint8_t)!heartbeat_on;
    }

    led_write(0U, heartbeat_on);
    led_write(1U, state_running);
    led_write(2U, (uint8_t)(now < rx_pulse_until));
    led_write(3U, state_error);

    vTaskDelayUntil(&last_wake, pdMS_TO_TICKS(50U));
}
```

### 9.6 LVGL 显示驱动注册

LVGL 通过注册 `flush_cb` 回调实现底层屏幕刷新。本工程使用 2 行显示缓冲，在保证正常刷新的同时控制 SRAM 占用：

```c
#define LVGL_BUFFER_LINES        2U

static lv_disp_draw_buf_t draw_buf;
static lv_color_t lvgl_buf[LCD_HOR_RES * LVGL_BUFFER_LINES];
```

显示驱动注册：

```c
static void lvgl_port_init(void)
{
    static lv_disp_drv_t disp_drv;

    lv_disp_draw_buf_init(&draw_buf, lvgl_buf, NULL, LCD_HOR_RES * LVGL_BUFFER_LINES);

    lv_disp_drv_init(&disp_drv);
    disp_drv.hor_res = LCD_HOR_RES;
    disp_drv.ver_res = LCD_VER_RES;
    disp_drv.flush_cb = lvgl_flush_cb;
    disp_drv.draw_buf = &draw_buf;
    (void)lv_disp_drv_register(&disp_drv);
}
```

刷新回调将 LVGL 输出的 RGB565 像素数据写入 LCD：

```c
static void lvgl_flush_cb(lv_disp_drv_t *disp_drv, const lv_area_t *area, lv_color_t *color_p)
{
    if((area->x2 < 0) || (area->y2 < 0) ||
       (area->x1 >= (lv_coord_t)LCD_HOR_RES) || (area->y1 >= (lv_coord_t)LCD_VER_RES)) {
        lv_disp_flush_ready(disp_drv);
        return;
    }

    lcd_flush_area((uint16_t)area->x1,
                   (uint16_t)area->y1,
                   (uint16_t)area->x2,
                   (uint16_t)area->y2,
                   (const uint16_t *)color_p);

    lv_disp_flush_ready(disp_drv);
}
```

### 9.7 LCD 区域刷新函数

官方 LCD 例程提供点、线、色块等裸机显示接口。为适配 LVGL，新增区域刷新函数 `lcd_flush_area()`：

```c
void lcd_flush_area(uint16_t x_start, uint16_t y_start, uint16_t x_end, uint16_t y_end, const uint16_t *color_p)
{
    uint32_t count;
    uint16_t color;

    count = ((uint32_t)x_end - x_start + 1U) * ((uint32_t)y_end - y_start + 1U);
    lcd_set_region(x_start, y_start, x_end, y_end);
    LCD_RS_SET;
    LCD_CS_CLR;

    while(count > 0U) {
        color = *color_p++;
        spi_write_byte(SPI1, (uint8_t)(color >> 8));
        spi_write_byte(SPI1, (uint8_t)color);
        count--;
    }

    LCD_CS_SET;
}
```

此函数是 LVGL 移植的关键点。LVGL 仅告知”需要刷新的区域及像素数据”，具体的 LCD 地址窗口设置和 SPI RGB565 写入由 BSP 层完成。

### 9.9 显示问题修复

### 10.2 DHT11 引脚规划

DHT11 DATA 引脚规划为 `PB0`：

| DHT11 引脚 | GD32C231C-EVAL 连接 |
| --- | --- |
| VCC | 3V3 |
| GND | GND |
| DATA | PB0 |

![DHT11 接线实物图](https://raw.githubusercontent.com/318973613/gd32c231c-claude-monitor/master/doc/a053b7c2858af7cb1dee0f85003a6e07.jpg)

- LCD 已占用 SPI1 相关引脚（PB13/PB14/PB15）及控制引脚（PA8/PC7/PC6）

- PB0 不与上述外设冲突

### 10.5 单总线时序

DHT11 通信流程：

1. MCU 拉低 DATA 至少 18 ms（启动信号）
2. MCU 释放 DATA
3. DHT11 响应低电平约 80 μs，再拉高约 80 μs
4. DHT11 输出 40 bit 数据
5. 每 bit 先低电平约 50 μs，高电平宽度决定 0 或 1

启动信号使用 FreeRTOS 延时：

```c
dht11_drive_low();
vTaskDelay(pdMS_TO_TICKS(DHT11_START_MS));
```

数据采样阶段使用短临界区保护：

```c
taskENTER_CRITICAL();

dht11_release_line();
dht11_delay_us(DHT11_RELEASE_US);

/* response and 40-bit sample */

taskEXIT_CRITICAL();
```

> **说明**：DHT11 的 0/1 判断依赖 26 μs 和 70 μs 高电平差异。若采样过程被 SysTick 或其他中断打断，可能导致误判。

### 10.6 数据采样优化

初始方案采用”统计高电平持续时间”判断 0/1，理论上可行，但在 Cortex-M23 + 编译优化后，空循环延时不稳定，易将 70 μs 高电平误判为短脉冲，导致 40 bit 全部读为 0。

DHT11 校验规则：

```text
checksum = 湿度整数 + 湿度小数 + 温度整数 + 温度小数
```

若读取到 `00 00 00 00 00`，校验和恰好为 0，旧逻辑误判为 `OK`，屏幕显示 `0.0 C / 0.0 %`。

优化方案：每个 bit 高电平开始后延时约 40 μs 再采样：

```c
dht11_delay_us(DHT11_BIT_SAMPLE_US);

raw[bit / 8U] <<= 1U;
if(0U != dht11_read_pin()) {
    raw[bit / 8U] |= 1U;
}
```

判断逻辑：

- 40 μs 后 DATA 已变低 → 短高电平 → bit = 0
- 40 μs 后 DATA 仍为高 → 长高电平 → bit = 1

新增全 0 数据保护：

```c
if((0U == raw[0]) && (0U == raw[1]) && (0U == raw[2]) && (0U == raw[3]) && (0U == raw[4])) {
    return DHT11_ERROR_ZERO_DATA;
}
```

屏幕显示 `ZERO` 而非误报有效温湿度，便于排查时序或接线问题。

### 10.7 微秒延时实现

普通空循环延时受编译优化影响明显。本工程采用基于 `SysTick->VAL` 的微秒延时，利用 FreeRTOS 已使用的 SysTick 定时器当前递减计数值估算经过的 CPU tick：

```c
static void dht11_delay_us(uint32_t us)
{
    uint32_t ticks_per_us;
    uint32_t target_ticks;
    uint32_t elapsed_ticks = 0U;
    uint32_t reload;
    uint32_t last;
    uint32_t now;

    ticks_per_us = SystemCoreClock / 1000000U;
    if(0U == ticks_per_us) {
        ticks_per_us = 1U;
    }

    target_ticks = us * ticks_per_us;
    reload = SysTick->LOAD;
    last = SysTick->VAL;

    while(elapsed_ticks < target_ticks) {
        now = SysTick->VAL;
        if(last >= now) {
            elapsed_ticks += last - now;
        } else {
            elapsed_ticks += last + (reload - now) + 1U;
        }
        last = now;
    }
}
```

> **技术要点**：DHT11 单总线时序对延时精度敏感。在 RTOS 环境下，需明确哪些位置可让出 CPU，哪些位置必须短时间保持时序稳定。

### 10.8 DHT11 采集任务

DHT11 任务每 2 秒采集一次，结果写入共享状态：

```c
static void dht11_task(void *argument)
{
    dht11_data_t data;
    dht11_status_t status;

    (void)argument;

    dht11_init();
    vTaskDelay(pdMS_TO_TICKS(1500U));

    while(1) {
        status = dht11_read(&data);

        taskENTER_CRITICAL();
        sensor_state.status = status;
        sensor_state.sample_count++;
        if(DHT11_OK == status) {
            sensor_state.data = data;
        } else {
            sensor_state.error_count++;
        }
        taskEXIT_CRITICAL();

        vTaskDelay(pdMS_TO_TICKS(DHT_SAMPLE_PERIOD_MS));
    }
}
```

`sample_count` 和 `error_count` 用于现场调试，可快速判断传感器是偶发错误还是持续错误。

## 12. 阶段六：Claude UI 资源外部 Flash 方案

完成 `FreeRTOS + LVGL + DHT11` 集成后，开发板已具备屏幕显示、传感器采集和串口通信能力。但在实现 Claude 风格界面时，面临一个现实问题：GD32C231C8 片内资源有限（64 KB Flash / 12 KB SRAM），若将图片、大字体、复杂 UI 控件全部置于内部 Flash 或 SRAM，工程资源将迅速耗尽。

本阶段采用更接近实际产品的方案：将固定 UI 资源预烧至板载外部 Flash，应用程序启动后从外部 Flash 读取资源显示。主程序仅负责逻辑处理、传感器数据采集、串口通信及动态数据更新，图片和界面素材交由外部 Flash 存储。

![外部 Flash 资源方案架构](https://raw.githubusercontent.com/318973613/gd32c231c-claude-monitor/master/doc/9fe9fc4ee24ee0e8a4e8d49a7ef7c392.jpeg)

### 12.1 方案选择依据

开发板片内 SRAM 有限，LVGL 阶段已显现资源压力。第 05 阶段加入 DHT11 后，静态 SRAM 占用已接近 11 KB。后续若继续增加串口、时钟、状态界面、图片或大字体，片内资源将难以承受。

初期尝试使用 LVGL label 构建 Claude 风格界面，虽可显示但效果有限，且对象数量增加后 LVGL heap 和 FreeRTOS heap 均趋于紧张。因此调整思路：固定不变的背景图、装饰图、文案资源不应占用 MCU 片内 Flash，更不应长期占用 SRAM。

本阶段最终采用的架构：

```text
PC 生成 UI 资源包
        ↓
资源烧录工程 07 写入 GD25Q16
        ↓
应用工程 08 启动后读取 GD25Q16
        ↓
LCD 显示背景 + 动态叠加时间 / DHT11 / 串口状态
```

方案优势：

- UI 图片容量可超过片内 Flash 限制
- 应用固件无需因更换背景图而频繁修改代码
- SRAM 仅需一行解码缓冲，无需整屏 framebuffer
- 后续接入 Claude 或其他上位机协议时，应用逻辑更清晰

LCD 同样使用 `SPI1`，仅片选和控制脚不同：

| 功能 | 引脚 |
| --- | --- |
| LCD CS | PA8 |
| LCD DC | PC7 |
| LCD RST | PC6 |
| LCD SPI | PB13 / PB14 / PB15 |

> **关键说明**：LCD 和 Flash 共用 `SPI1`，但 SPI 模式不同。LCD 驱动使用 `SPI_CK_PL_HIGH_PH_2EDGE`，GD25Q16 Flash 驱动使用 `SPI_CK_PL_LOW_PH_1EDGE`。两个驱动不可直接混用，每次切换设备需重新配置 SPI1 模式。

### 12.7 SPI1 总线切换处理

Flash 和 LCD 共用 SPI1，但 SPI 模式不同，这是本阶段的关键技术难点。解决方案是在 LCD 驱动中增加总线初始化接口：

```c
void lcd_bus_init(void)
{
    spi1_init();
}
```

`spi1_init()` 和 `spi_flash_init()` 中均先复位 SPI1：

```c
spi_i2s_deinit(SPI1);
```

每次切换设备时，SPI1 按当前设备需要的模式重新配置。

背景图绘制流程：

```text
切到 Flash 模式
读取一段 RLE 数据
解码到一行 line_buffer
切回 LCD 模式
将该行刷到 LCD
继续下一行
```

此方案虽非最快，但对 12 KB SRAM 的芯片稳定可靠。未申请整屏 framebuffer，仅保留一行 240 像素的 RGB565 缓冲：

```c
static uint16_t line_buffer[240];
```

占用约 480 字节 SRAM。

### 12.12 本阶段总结

本阶段验证的核心结论：GD32C231C（64 KB Flash / 12 KB SRAM）不能采用”大 MCU + 大 framebuffer”的 UI 方案。可行方式是将固定资源与动态覆盖层分离。

当前架构：

```text
资源生成脚本 generate_ui_assets.py
  ↓
生成 ui_assets.bin / ui_assets.c / 预览图
  ↓
第 07 工程将 CUI1 资源包写入 GD25Q16
  ↓
第 08 工程启动时从 GD25Q16 读取背景
  ↓
LCD 显示静态背景
  ↓
裸 LCD 小区域刷新时钟 / DHT11 / 计数器 / 状态头像 / 动画
```

工程职责划分：

- 第 07 工程：资源烧录器，负责将 UI 资源包可靠写入 GD25Q16 并读回校验
- 第 08 工程：实际应用，负责 FreeRTOS、DHT11、USART0 时钟、LED 灯效、LCD 显示及运行时小区域刷新

当前第 08 工程虽保留 LVGL 源码参与编译，但实际 UI 路径未调用 `lv_init()`、`lv_timer_handler()`、`lv_obj_*()` 或 `lv_label_*()`。此为有意保守处理：先稳定外部 Flash 资源、SPI1 模式切换、DHT11、串口和 LCD 刷新，再考虑是否重新引入 LVGL 部分能力。

关键经验：

1. 大图和固定素材存入 GD25Q16，片内 SRAM 仅保留一行 `line_buffer[240]` 和少量覆盖层缓冲
2. LCD 和 GD25Q16 共用 SPI1 时，读 Flash 前须调用 `spi_flash_init()`，画 LCD 前须调用 `lcd_bus_init()`，因两者 SPI 模式不同
3. 动态 UI 不必依赖完整 GUI 框架，240x320 小屏使用像素数字、矩形卡片和小区域动画更易控制内存和刷新范围
4. 动画应避免整块刷新，仅刷新变化小块可保持连贯性且不拖慢主循环

后续扩展方向：

1. 扩展 `CUI1` 资源包，加入更多状态图标或背景（idle、working、thinking、error）
2. 完善状态机，使未连接、等待命令、DHT11 错误、串口收到命令等状态对应不同图标
3. UI 和底层链路稳定后接入 Claude 通信协议，通信状态走 USART 或其他通道，固定视觉资源继续存外部 Flash，动态状态由第 08 工程小区域刷新

## 13. 阶段七：Windows 上位机与 Claude Code 状态联动

第 12 阶段完成后，开发板已经具备稳定的显示、动画、DHT11、LED 和串口协议能力。本阶段继续向“Claude 状态监视器”推进：在 PC 端实现一个 Windows 上位机，用于连接 Claude Code、串口和开发板 LCD，使 Claude 的工作状态、模型信息、上下文使用率、费用和权限审批状态能够显示到 GD32C231C-EVAL 上。

本阶段不是让开发板直接访问网络或调用 Claude API，而是采用更稳妥的三段式架构：

```text
Claude Code
  ↓ Hooks / statusLine
Windows 上位机
  ↓ USART0 115200 8N1
GD32C231C-EVAL LCD / LED / PA0 / PA4
```

这样设计的原因很直接：GD32C231C8 只有 64 KB Flash 和 12 KB SRAM，不适合直接承担 HTTPS、JSON API、证书、复杂 UI 以及模型状态解析。Claude Code 本身运行在 PC 上，上位机天然更适合处理 Hooks、statusLine、权限确认和串口转发，开发板只负责显示和交互。

### 13.5 PA0 / PA4 权限审批

本阶段将开发板按键用于 Claude 权限审批：

| 按键 | 开发板引脚 | 功能 |
| --- | --- | --- |
| PA0 | Wakeup | 同意 |
| PA4 | Userkey | 拒绝 |

当 Claude Code 触发需要审批的工具调用时，如果上位机处于“外部审批模式”，Claude 会等待上位机返回决定。此时可以通过三种方式审批：

1. 按开发板 PA0 同意
2. 按开发板 PA4 拒绝
3. 在上位机 GUI 中点击“同意 / 拒绝”

为了提升按键体验，固件端将按键扫描周期调整为 `10 ms`，上位机端增加了 2.5 秒预按缓存。也就是说，如果用户稍早按下 PA0/PA4，而审批请求马上到达，上位机会缓存这次按键并自动应用到最近的审批请求上，避免必须连续按很多次。

当外部审批模式关闭时，Claude 会使用原生终端确认。此时上位机和开发板会提示进入终端确认状态，开发板显示 `TERM 123`，表示需要回到 Claude Code 终端中按 `1/2/3`，PA0/PA4 不会控制这一次原生提示。

### 13.6 LED 状态设计

四个板载 LED 不再只是简单流水灯，而是作为辅助状态提示：

| 状态 | LED 效果 |
| --- | --- |
| 空闲 | 慢心跳 |
| 工作中 / 工具调用 | 流水灯 |
| Claude 原生终端确认 | 对角交替闪烁 |
| 等待外部审批 | 左右交替闪烁 |
| 已同意 | 全亮脉冲 |
| 已拒绝 | 单灯警示 |

LCD 可以显示详细信息，LED 则适合从远处快速判断状态。例如看到左右交替闪烁，就知道当前不是普通运行，而是需要用户审批。

## 15. 总结

本文完整记录了 GD32C231C-EVAL 开发板从开箱到实现 Claude 状态监视器的全过程。整个评测遵循"由浅入深、逐层递进"的原则，将一个复杂的嵌入式项目拆解为七个可独立验证的阶段。

### 15.1 技术路线回顾

| 阶段 | 核心内容 | 关键技术 |
|------|----------|----------|
| 阶段一 | GPIO 流水灯 | GPIO 输出、推挽配置、bit mask 控制 |
| 阶段二 | USART 串口链路 | USART0、中断接收、命令协议 |
| 阶段三 | FreeRTOS 移植 | Cortex-M23 port、任务划分、队列通信 |
| 阶段四 | LVGL 屏幕显示 | SPI LCD、flush callback、tick 接入 |
| 阶段五 | DHT11 传感器 | 单总线时序、微秒延时、数据校验 |
| 阶段六 | 外部 Flash UI | GD25Q16、SPI1 模式切换、RLE 压缩 |
| 阶段七 | Claude Code 联动 | Windows 上位机、Hooks、权限审批 |

### 15.2 核心经验总结

**1. 资源受限下的工程策略**

GD32C231C8 的 64 KB Flash 和 12 KB SRAM 决定了不能采用"大 MCU + 大 framebuffer"的方案。本项目的解决思路是：

- 大图和固定素材存入外部 Flash GD25Q16
- 片内 SRAM 仅保留一行解码缓冲（480 字节）
- 动态 UI 采用裸 LCD 矩形和像素数字，不依赖完整 GUI 框架

**2. 多任务架构设计**

FreeRTOS 的引入使程序结构从"主循环轮询"升级为"任务驱动"：

- LED 任务：低频状态刷新
- 串口任务：高优先级命令响应
- GUI 任务：周期性显示更新
- 传感器任务：定时数据采集

各任务通过队列和临界区共享数据，职责清晰，便于维护和扩展。

**3. 外设共用 SPI1 的处理**

LCD 和 GD25Q16 共用 SPI1，但 SPI 模式不同。解决方案是在每次切换设备前调用对应的初始化函数（`lcd_bus_init()` 或 `spi_flash_init()`），通过 `spi_i2s_deinit()` 复位后重新配置。

**4. PC 与嵌入式的分工**

本项目采用三段式架构：

- Claude Code：运行在 PC，负责 AI 推理和 Hooks
- Windows 上位机：运行在 PC，负责协议转换和串口转发
- GD32C231C-EVAL：负责显示、交互和传感器采集

这种分工让 MCU 只承担适合其资源能力的任务，避免了在小芯片上强行实现 HTTPS、JSON 解析等复杂功能。

### 15.3 项目成果

经过七个阶段的迭代，最终实现的功能：

- **LCD 显示**：外部 Flash 背景 + 裸 LCD 覆盖层，显示时间、温湿度、Claude 状态
- **状态动画**：空闲小人（耳机+音符）、工作小人（电脑+代码块+星星）
- **DHT11 温湿度**：实时采集和显示
- **LED 状态指示**：心跳、运行、审批、错误等多种灯效
- **串口协议**：支持 Claude 状态、时钟设置、模型信息等命令
- **权限审批**：PA0 同意、PA4 拒绝，支持预按缓存
- **Windows 上位机**：Claude Code Hooks 集成，图形化状态显示
